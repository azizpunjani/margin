margin▍

AI code review in your editor's margin — powered by one JSONL file.

Review a diff in Neovim or a zero-dep TUI. Leave a comment inline; an AI agent answers threaded in place — and can edit the code when you ask. No server, no sockets, no browser tab.

Features

Built around one thesis: speed. Appending a record takes ~1 ms, the watcher wakes in 0.2 s, rendering takes <100 ms. The only real wait in the loop is model inference.

💬

Inline threaded comments

Comment on any line; replies interleave in timestamp order under the code: 💬 you, 🤖 AI, 👤 your follow-ups.

✏️

AI code edits from comments

"Rename this", "handle null here" — the agent makes the edit, marks the reply "edit":true, and your buffer reloads with the change.

💡

First-pass observations

When a review starts, the bundled skill scans the diff once and flags possible bugs or risks as 💡 threads (a handful max, no style nits).

▍

Streaming replies

Long answers stream in as reply-chunk records with a typing cursor (▌) until the final reply supersedes them.

↕️

Visual-range comments

Select lines in visual mode, hit <leader>mc — the thread shows [L26–31] and the AI reasons about the whole span.

✓

Resolve & archive

<leader>mx resolves the thread at the cursor; :MarginResolveAll clears the board. Resolved threads stop rendering.

📂

File picker with counts

<leader>mf lists the files under review with their comment counts inline; jump straight to a file's tab.

☷

Diff review vs any base

:MarginReview [base] opens diff tabs — base version left (readonly), working file right. Files new at base open plain.

👁️

Presence priming

When your cursor dwells ~2 s, nvim writes the location to presence.json; the agent can pre-read that code so answers start from context.

⚡

Zero-dep compiled TUI

A single compiled binary (Bun + raw ANSI): view a diff, comment, and read replies without an editor. Threads, ranges, and resolve are Neovim-only today.

📄

An open, file-based protocol

Everything is append-only JSONL in .margin/review.jsonl. Anything that can read and append a file can join the conversation.

How it works

The transport is a file — on purpose

No server, no sockets, no browser. Editor and agent meet in one append-only JSONL file at the repo root — plus an ephemeral presence sidecar. Do the math before calling it slow: appending is ~1 ms and the render wakes in under 100 ms, while the model's answer takes seconds — transport is a rounding error in this loop, and a socket would optimize the part that doesn't matter. What the file buys instead: cat-level debuggability, a replayable record of the whole review, crash-safe append-only writes, and zero coupling — any editor or agent that can tail and append can join. git, maildir, SQLite: the boring option has good company.

01

Editor appends

You comment on a line; Neovim (or the TUI) appends a comment record to .margin/review.jsonl. ~1 ms.

02

Agent watches & answers

The AI's watcher wakes in 0.2 s, reads the code around the line, and appends its reply — or edits the file first.

→
03

Editor renders live

The editor watches the same file and re-renders threads under the code in <100 ms. The only real wait is inference.

# .margin/review.jsonl — one JSON object per line, append-only
{"type":"comment",       "id":"c1", "file":"src/pay.ts", "line":26, "endLine":31, "side":"new", "text":"what is this code about?"}
{"type":"reply-chunk",   "replyTo":"c1", "text":"reading pay.ts:26… "}        # streamed; rendered with a ▌ cursor
{"type":"reply",         "replyTo":"c1", "text":"Stripe's standard test card…"}  # final; supersedes the chunks
{"type":"user-reply",    "replyTo":"c1", "text":"swap it for a Mastercard"}     # reopens the thread
{"type":"reply",         "replyTo":"c1", "text":"Swapped to 5555…4444", "edit":true}  # the AI changed the code
{"type":"resolve",       "ids":["c1"]}                                     # archived; stops rendering
{"type":"review-request","files":["src/pay.ts"], "base":"HEAD"}            # AI asks YOU to review its changes

# .margin/presence.json — ephemeral sidecar, overwritten in place (not part of the log)
{"file":"src/pay.ts", "line":27, "ts":1730000000000}   # where your cursor is dwelling right now

Integrations

Neovim and the TUI ship in the box. On the AI side, Claude Code works out of the box via the bundled skill — but the protocol is just a file, so any agent can play.

Neovim plugin

Zero-dependency, native vim.pack-style plugin — just prepend margin/nvim to your runtimepath. Comment from normal buffers while editing; threads render as virt_lines under the code, and :checktime reloads AI edits into your buffers.

Key / commandAction
<leader>mcComment on current line (visual mode: whole range)
<leader>mrReply within the thread at / nearest above the cursor
<leader>mxResolve thread (:MarginResolveAll for all)
<leader>mn / mpJump next / prev thread in buffer
<leader>mfFile picker with comment counts
<leader>mqAll threads to quickfix
:MarginReview [base]Diff tabs vs a base rev or the AI's latest review-request; new-at-base files open plain
:MarginClear / :MarginShowHide / re-show threads in buffer

Claude Code

The bundled margin-review skill makes Claude Code a review partner: it watches the review file, acks new comments, streams answers, edits code on request, pre-reads code near your cursor via presence.json, and appends review-requests for its own changes.

Bring your own agent

The protocol is a file. An integration is a tail and an append:

tail -f .margin/review.jsonl | while read -r rec; do
  # when rec is a "comment": think, then answer with one append
  echo '{"type":"reply","replyTo":"c1","ts":1730000000000,"text":"…"}' >> .margin/review.jsonl
done

Terminal multiplexers

When the AI wants you to review its changes, it can auto-open an nvim split beside your session (herdr panes supported out of the box; any multiplexer with a split-and-run command — tmux, wezterm — is a few lines in the skill). No multiplexer? It degrades gracefully — you just run nvim +MarginReview yourself.

TUI — no editor required

A compiled, zero-dependency terminal UI: margin diffs your working tree (any git-diff args pass through), c comments, n/N jump threads, replies render live. Review from any shell.

Install

$ git clone https://github.com/azizpunjani/margin
$ cd margin && ./install.sh
  • Builds the TUI to a single compiled binary and links it at ~/.local/bin/margin.
  • Installs the Claude Code skill (margin-review) into ~/.claude/skills/.
  • Prints the Neovim loader snippet — one line: vim.opt.rtp:prepend('…/margin/nvim').
  • Idempotent — safe to re-run after git pull.

Prerequisites: bun is required to build the TUI binary. For a Neovim-only setup you don't need to build anything — skip the script and just add margin/nvim to your runtimepath.
